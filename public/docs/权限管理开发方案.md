# 权限管理开发方案

更新时间：2026-06-16

---

## 一、目标

**核心理念：权限以 permission_code 为最小单元。** 角色、用户最终关联的都是 permission_code；菜单只是 code 的可视化勾选载体——勾选菜单只是输入手段，最终保存进系统的是"权限 code"，不是"菜单关系"。

基于 Casbin 构建权限系统，覆盖三个层次：

```
Layer 1: API 权限     — 能不能访问这个接口？
Layer 2: 行级数据权限  — 能看见哪些数据行？（时间范围、部门范围等）
Layer 3: 字段级权限   — 能看见哪些数据列？（敏感字段屏蔽）
```

> 实现上 Layer 2（行级过滤）与 Layer 3（字段屏蔽）合并到同一张 `sys_rule` 表，由 `pkg/datarule` 的 GORM 插件统一处理，不单列组件。

权限关系全在 Casbin：给角色/用户分配菜单时，收集菜单的 permission_code 去重后写入 Casbin，存的是"角色-权限""用户-权限"，而不是"角色-菜单""用户-菜单"。

---

## Casbin 管什么，新增表管什么

Casbin 只做一件事：**判断"谁(sub)能不能对什么资源(obj)做什么操作(act)"**，返回 true/false。

以下场景 Casbin **不做**，需要新增表和逻辑：

| 场景 | Casbin 为什么做不了 | 谁来处理 |
|------|--------------------|---------|
| 订单列表只显示最近 30 天 | Casbin 不参与 SQL 构建，不做行级过滤 | `sys_rule` 表 + GORM 插件 |
| 金额字段对某些人不可见 | Casbin 不知道"字段"这个概念 | `sys_rule` 表 + GORM Omit |
| 某个接口无需登录即可访问 | Casbin 前置于 JWT，无法表达"免登录" | 中间件路由分组 / 白名单 |
| 角色需要一个展示名（role_name） | Casbin 是策略引擎，存不了实体属性 | `sys_role` 表 |

## 二、整体架构

```
请求 → JWT 认证 → 获取 user_id
    │
    ├── Layer 1: API 权限（Casbin）
    │   请求 URL + Method
    │   → Casbin enforce(user_id, URL_PATH, METHOD)
    │   → Casbin 中 user_id 同时拥有角色权限 + 直接分配的用户权限
    │   → 通过 / 拒绝
    │
    └── Layer 2: 数据权限（GORM Plugin）
         业务 GORM 查询
         → GORM BeforeQuery 拦截器
         → 检测数据域 domain
         → 加载 sys_rule JSON 规则
         → 翻译为 GORM Where / Omit
         → 执行过滤后的查询
```

### 3.1 API 权限校验流

```
┌─────────────┐
│  HTTP 请求   │  GET /api/admin/list
└──────┬──────┘
       │
       ▼
┌─────────────────────┐

└──────┬──────────────┘
       │
       ▼
┌──────────────────────────────┐
│  Casbin 权限校验              │
│  enforce(user_id, URL, METHOD)│
│                              │
│  Casbin 策略中 user_id 直接  │
│  对应用户权限，也通过 g 映射 │
│  继承角色权限，无需分开校验  │
│                              │
│  ├─ 通过 ──→ 继续            │
│  └─ 拒绝 ──→ 403             │

```

### 3.2 权限分配流（菜单 → Casbin）

```
┌────────────────────┐
│  前端勾选菜单树       │
│  type=1 目录(不可勾选)│
│  type=2 菜单(列表页)  │
│  type=3 按钮(CRUD)    │
└──────┬─────────────┘
       │
       ▼
┌──────────────────────────────┐
│  后端收集 permission_code    │
│  遍历勾选的菜单 → 取 permission_code │
│  → 去重                      │
└──────┬───────────────────────┘
       │
       ▼
┌──────────────────────────────┐
│  写入 Casbin                 │
│  p, target_id, path, method, code │
│  角色: target_id = role_code │
│  用户: target_id = user_id   │
└──────────────────────────────┘
```

### 3.3 数据权限校验流

```
┌────────────────┐
│  业务查询        │  db.Model(&Order{}).Find(&orders)
└───────┬────────┘
        │
        ▼
┌─────────────────────────────┐
│  GORM BeforeQuery 插件       │
│                              │

│     WHERE domain = 'ORDER'   │

│       )                     │
│  4. 翻译为 GORM 查询条件      │
│     ──→ Where / Omit        │
│  5. 注入到 GORM Statement    │
└───────┬─────────────────────┘
        │
        ▼
┌────────────────┐
│  执行带过滤的查询 │  SELECT id,title,... FROM orders WHERE ... AND (dept_id IN (...))
└────────────────┘
```

## 四、数据库表结构

### 4.1 已有表

| 表名 | 用途 | 说明 |
|------|------|------|
| `sys_admin` | 管理员用户 | 已有 |
| `sys_menus` | 菜单树（permission_code 的可视化勾选载体） | type: 1=目录 2=菜单(列表页) 3=按钮(CRUD) 4=外链/iframe。type=2,3 必含 permission_code；type=1,4 不需要 |
| `sys_casbin_rule` | Casbin 策略（权限关系全在这） | v0=用户ID 或 角色 role_code, v1=URL, v2=方法, v3=code，自动管理 |

### 4.2 权限分配核心逻辑

不需要额外的菜单-权限绑定表或用户例外表。整个流程是：

```
sys_menus(type=2,3).permission_code  →  勾选菜单  →  收集 permission_code  →  去重  →  sys_casbin_rule
```
- type=1 目录：仅前端容器，不参与权限收集，不写入 Casbin
- type=4 外链/iframe：不调后端 API，不参与权限收集

- 给角色分配菜单 → `p, role_code, path, method, code`
- 给用户直接分配菜单 → `p, user_id, path, method, code`
- Casbin enforce 时，`user_id=p.sub` 直接命中用户策略，同时通过 g 映射 `g, user_id, role_code` 命中角色策略
- 不需要 sys_menu_permission（菜单已自带 permission_code）
- 不需要 sys_user_permission_exception（想给用户多一个权限就多分配一个菜单）
- **不需要 sys_admin_role**：用户↔角色关系由 Casbin g 映射承载（`g, user_id, role_code`），不建关联表
- **需要 sys_role（只存角色元信息）**：角色作为"权限组"需要展示名（role_name）、状态、排序等，Casbin 存不了这些实体属性；权限关系（角色含哪些 code）仍由 Casbin p 策略承载。sys_role 通过 role_code 与 Casbin 关联

**sys_role** — 角色表（仅存元信息，权限关系在 Casbin）

```sql

  `role_name` varchar(100) CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci NOT NULL COMMENT '角色名称（前端展示）',
  `status` tinyint NOT NULL DEFAULT '1' COMMENT '状态: 0=禁用 1=启用',
  `sort_order` int NOT NULL DEFAULT '0' COMMENT '排序',
  `remark` varchar(200) CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci DEFAULT NULL COMMENT '备注',
  `create_by` bigint unsigned DEFAULT NULL COMMENT '创建人ID',
  `create_time` datetime(3) DEFAULT NULL COMMENT '创建时间',
  `update_by` bigint unsigned DEFAULT NULL COMMENT '更新人ID',
  `update_time` datetime(3) DEFAULT NULL COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_role_code` (`role_code`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='角色表（仅存元信息，权限关系在 Casbin）';
```

**解决的问题（数据权限）**：统一控制数据访问权限，包含两种能力：
- **字段屏蔽**：用户不能查看某些字段（如金额、利润率）
- **内容过滤**：用户不能查看包含特定内容的行（如标题含"折扣"）

规则模板存一次，通过分配表绑定到多个角色或用户。

```sql
CREATE TABLE `sys_rule` (
  `id` bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `rule_name` varchar(100) NOT NULL COMMENT '规则名称',
  `domain` varchar(50) NOT NULL COMMENT '数据域标识（ORDERS / NOTICE / ADMIN 等）',
  `config` json NOT NULL COMMENT '规则配置JSON（含 omit_fields + condition_groups）',
  `status` tinyint NOT NULL DEFAULT '1' COMMENT '状态: 0=禁用 1=启用',
  `remark` varchar(200) DEFAULT NULL COMMENT '备注',
  `create_by` bigint unsigned DEFAULT NULL COMMENT '创建人ID',
  `create_time` datetime(3) DEFAULT NULL COMMENT '创建时间',
  `update_by` bigint unsigned DEFAULT NULL COMMENT '更新人ID',
  `update_time` datetime(3) DEFAULT NULL COMMENT '更新时间',
  PRIMARY KEY (`id`),
  KEY `idx_domain` (`domain`),
  KEY `idx_status` (`status`)
) ENGINE=InnoDB COMMENT='数据权限规则表';
```

**sys_rule_assignment** — 规则分配

```sql
CREATE TABLE `sys_rule_assignment` (
  `id` bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `rule_id` bigint unsigned NOT NULL COMMENT '规则ID',
  `target_type` tinyint NOT NULL COMMENT '目标类型: 1=角色 2=用户',
  `target_id` bigint unsigned NOT NULL COMMENT '目标ID（role_id 或 admin_id）',
  `create_by` bigint unsigned DEFAULT NULL,
  `create_time` datetime(3) DEFAULT NULL,
  PRIMARY KEY (`id`),
  KEY `idx_rule_id` (`rule_id`),
  KEY `idx_target` (`target_type`, `target_id`),
  CONSTRAINT `fk_rule_assign` FOREIGN KEY (`rule_id`) REFERENCES `sys_rule` (`id`) ON DELETE CASCADE
) ENGINE=InnoDB COMMENT='规则分配表';
```

##### config JSON 格式

config 字段统一包含 `omit_fields`（字段屏蔽）和 `condition_groups`（内容过滤条件组），两者可同时存在，也可只用一个。

每组有自己的 `logic`（AND/OR），组之间默认 AND 连接。通过多组组合，可以实现复杂的过滤逻辑，避免全 AND 把所有数据都筛掉。

```json
{
  "omit_fields": ["amount", "profit_rate"],
  "condition_groups": [
    {
      "logic": "AND",
      "conditions": [
        { "field": "dept_id", "op": "EQ", "value": "user.dept_id", "desc": "只能看本部门" },
        { "field": "status", "op": "NOT_IN", "value": ["deleted"], "desc": "不能看已删除" }
      ]
    },
    {
      "logic": "OR",
      "conditions": [
        { "field": "title", "op": "NOT_LIKE", "value": "%折扣%", "desc": "标题不能含折扣" },
        { "field": "title", "op": "NOT_LIKE", "value": "%特价%", "desc": "标题不能含特价" }
      ]
    }
  ]
}
```

生成的 SQL：

```sql
SELECT id, title, status, dept_id   -- amount 被 Omit
FROM orders
WHERE (dept_id = 用户部门 AND status != 'deleted')
  AND (title NOT LIKE '%折扣%' OR title NOT LIKE '%特价%')
```

支持的 op 操作符：`EQ`, `NEQ`, `GT`, `GTE`, `LT`, `LTE`, `IN`, `NOT_IN`, `LIKE`, `NOT_LIKE`, `BETWEEN`

值引用：`user.*` 前缀表示从用户上下文取动态值（如 `user.dept_id`）；直接写值表示固定值。

##### 前端交互逻辑

新建规则时，前端展示规则构建器：

```
新建数据规则

  数据域: [订单管理 ORDERS]  ▼
  规则名称: [客服权限限制]

  ┌─ 字段屏蔽 ─────────────────────┐
  │  ☑ amount (订单金额)            │
  │  ☑ profit_rate (利润率)         │
  │  ☐ cost_price (成本价)          │
  └────────────────────────────────┘

  ┌─ 内容过滤 ────────────────────────────┐
  │  [+ 添加条件组]                         │
  │                                        │
  │  组 1 [AND ▼]                          │
  │    字段 [dept_id ▼]  操作 [=     ▼]  值 [自己部门        ]  [删除] │
  │    字段 [status  ▼]  操作 [!=    ▼]  值 [deleted         ]  [删除] │
  │                                        │
  │  组 2 [OR ▼]                           │
  │    字段 [title   ▼]  操作 [NOT_LIKE ▼]  值 [%折扣%          ]  [删除] │
  │    字段 [title   ▼]  操作 [NOT_LIKE ▼]  值 [%特价%          ]  [删除] │
  └────────────────────────────────────────┘

  应用于: [角色 ▼] [管理员]
```

每组可选 AND/OR，组之间固定 AND 连接。

**操作符说明：** 添加条件时，操作符默认 `=`（等于）。点击下拉可切换为 `!=`、`>`、`>=`、`<`、`<=`、`IN`、`NOT_IN`、`LIKE`、`NOT_LIKE`、`BETWEEN`。其中 `LIKE`/`NOT_LIKE` 用于模糊搜索，值里用 `%` 匹配任意字符（如 `%折扣%` 表示标题包含"折扣"）。

##### 使用示例

```sql
-- 建一条规则：隐藏金额字段 + 过滤含折扣的内容
INSERT INTO sys_rule (rule_name, domain, config, status)
VALUES (
  '客服权限',
  'ORDERS',
  '{"omit_fields":["amount","profit_rate"],"condition_groups":[{"logic":"AND","conditions":[{"field":"title","op":"NOT_LIKE","value":"%折扣%"}]}]}',
  1
);

-- 分配给用户 3
INSERT INTO sys_rule_assignment (rule_id, target_type, target_id)
VALUES (1, 2, 3);

-- 也分配给角色 2
INSERT INTO sys_rule_assignment (rule_id, target_type, target_id)
VALUES (1, 1, 2);
```

## 五、前端权限展示设计

### 5.1 菜单-权限树（权限分配界面）

所有菜单按 type 分层展示，菜单自带 permission_code：

```
type 1 = 目录（分组，无页面，不可勾选，只做容器，无需 permission_code）
type 2 = 菜单（列表页，可导航，可勾选，必须有 permission_code）
type 3 = 按钮（CRUD，处于对应菜单下，可勾选，必须有 permission_code）
type 4 = 外链/iframe（外部链接，无需 permission_code）

示例：
系统管理 (type=1, 目录)
├── ☐ 用户管理 (type=2, 菜单 → path: /admin/user, perm_code: admin:list)
│   ├── ☐ 新增 (type=3, 按钮 → perm_code: admin:create)
│   ├── ☐ 编辑 (type=3, 按钮 → perm_code: admin:update)
│   └── ☐ 删除 (type=3, 按钮 → perm_code: admin:delete)
├── ☐ 角色管理 (type=2, 菜单 → path: /admin/role, perm_code: role:list)
└── ☐ 操作日志 (type=2, 菜单 → path: /admin/log, perm_code: log:list)
```

无需 sys_menu_permission 表过滤 — type=2,3 自带 permission_code，type=1,4 不参与权限。

### 5.2 权限落地方案（菜单 → Casbin）

```
前端：勾选菜单树
  │
  ▼
后端：
  1. 遍历勾选的菜单（只处理 type=2 和 type=3）
  2. 收集 permission_code（去重）
  3. 查 path 和 method（菜单自带或从路由注册获取）
  4. 写入 Casbin: p, target_id, path, method, code
```

Casbin 策略示例（sys_casbin_rule）：

```
ptype  v0      v1                   v2    v3
p      admin   /api/admin/list      GET   admin:list     (角色 admin 的权限)
p      admin   /api/admin/create    POST  admin:create
p      3       /api/admin/list      GET   admin:list     (用户 3 的直接权限)
g      3       admin                                                   (用户 3 属于角色 admin)
```

**核心原则**：
- **URL（v1）+ 方法（v2）**：参与 enforce，用于请求时鉴权
- **code（v3）**：不参与 enforce，只做关联 key，用于菜单渲染和角色/用户权限绑定
- **菜单只是 code 的勾选载体**：勾选菜单只是输入手段，后端收集的是 permission_code；鉴权时完全不查 sys_menus
- **sub 标识约定**：用户侧 sub = `user_id`（数字字符串），角色侧 sub = `role_code`（字符串，如 `admin`）；角色元信息存 `sys_role`，用户↔角色关系通过 Casbin `g, user_id, role_code` 关联

### 5.3 用户渲染菜单树

```
用户登录
  → 从 Casbin 获取用户所有 permission_code
     → 用户直接分配的 p 策略（sub=user_id）
     → 用户所属角色的 p 策略（通过 g 映射）
     → 收集所有 v3（code）去重
  → 拿 code 列表查 sys_menus
  → 组装树（按 type 分层：type=1 目录, type=2 菜单, type=3 按钮）
  → 返回前端渲染
```

### 5.4 用户额外权限

用户管理 → 选择用户 → 直接分配菜单（同角色分配界面）：
- 给用户直接勾选菜单 → 写入 Casbin `p, user_id, path, method, code`
- 想给用户多权限就多分配菜单，想少权限就少分配 — 不需要"例外"概念

## 六、Casbin 策略模型

### 6.1 模型定义

Casbin 模型改为 4 字段 `p = sub, obj, act, code`，code 不参与 matcher：

```
[request_definition]
r = sub, obj, act

[policy_definition]
p = sub, obj, act, code

[role_definition]
g = _, _

[policy_effect]
e = some(where (p.eft == allow))

[matchers]
m = g(r.sub, p.sub) && r.obj == p.obj && r.act == p.act
```

matcher 不变，code 不参与 enforce。

### 6.2 Policy 策略

```
# 角色权限：p, role_code, URL, METHOD, code
p, admin, /api/admin/list,   GET,  admin:list
p, admin, /api/admin/create, POST, admin:create
p, editor, /api/admin/list,  GET,  admin:list

# 用户↔角色关联：直接写入 Casbin g 策略（无独立角色关联表）
g, 1, admin
g, 2, editor
```

### 6.3 中间件实现

```
obj := c.Request.URL.Path           // "/api/admin/list"
act := c.Request.Method              // "GET"
sub := strconv.FormatInt(userID, 10)
ok, err := enforcer.Enforce(sub, obj, act)
```

不需要查任何映射表，URL + Method 直接命中策略。code（v3）只用在需要关联菜单、渲染权限树等场景。

### 6.4 permission_code 的两层分工

- **分配/展示层**（权限管理界面、前端）：code 是核心。角色-权限、用户-权限都以 code 为单位，勾选菜单本质是在选 code。
- **执行层**（Casbin 鉴权）：按请求的 path+method 判定（`Enforce(user_id, URL, Method)`），code 不参与 enforce，只做关联键。

一个 code ↔ 一个 (path, method) 一一对应：code 是"权限叫什么"，path+method 是"权限管哪个接口"。分配时按 code 想，执行时按接口判，无需在请求时做 URL→code 反查。

## 七、pkg 层新增组件

### pkg/datarule — 数据权限引擎

```
pkg/datarule/
├── datarule.go           # 组件入口 Init / Close / Ready
├── types.go              # 规则相关类型定义
├── engine.go             # 加载规则、匹配用户、缓存
├── injector.go           # 条件注入：condition_groups → GORM Where
├── omit.go               # 字段排除：omit_fields → GORM Omit
├── domain.go             # 数据域注册：表名 ↔ domain 显式映射
└── provider/
    └── gorm.go           # GORM 插件（BeforeQuery 拦截器）
```

#### 核心实现

**数据加载流程（engine.go）：**

```
用户查询某张表时：
  1. 确定 domain（通过表名 → 显式注册的映射）
  2. 查 sys_rule_assignment，找到该用户/角色被分配了哪些规则：
     SELECT rule_id FROM sys_rule_assignment
     WHERE (target_type=1 AND target_id IN role_ids)
        OR (target_type=2 AND target_id = admin_id)
  3. 根据 rule_id 加载规则模板：
     SELECT config FROM sys_rule WHERE id IN (rule_ids)
  4. 缓存结果
```

**GORM 插件（provider/gorm.go）：**

```go
type DataRulePlugin struct{}

func (p *DataRulePlugin) Name() string {
    return "DataRulePlugin"
}

func (p *DataRulePlugin) Initialize(db *gorm.DB) error {
    return db.Callback().Query().Before("gorm:query").
        Register("data_rule:inject", p.inject)
}

func (p *DataRulePlugin) inject(db *gorm.DB) {
    ctx := db.Statement.Context
    if ctx == nil {
        return
    }
    adminID, _ := ctx.Value("admin_id").(int64)
    roleIDs, _ := ctx.Value("role_ids").([]int64)
    if adminID == 0 {
        return
    }

    domain := resolveDomain(db.Statement.Table)

    // 查分配表 → 取 rule_id 列表 → 加载统一规则 config
    rules := engine.LoadRules(domain, adminID, roleIDs)
    for _, rule := range rules {
        // 字段屏蔽
        if len(rule.Config.OmitFields) > 0 {
            db.Omit(rule.Config.OmitFields...)
        }
        // 内容过滤：按条件组处理，每组有自己的 logic
        for _, group := range rule.Config.ConditionGroups {
            if len(group.Conditions) > 0 {
                injector.ApplyGroup(db, group)
            }
        }
    }
}
```

一条规则同时处理字段屏蔽和内容过滤，不需要分两张表查两次。

#### 安全与实现约定（必守）

1. **ctx 缺失必须拒绝，不得静默放行**：示例中 `adminID == 0` 时 `return` 仅表示"非业务查询"（如内部初始化）。正式实现中，命中已注册 domain 的查询若取不到身份信息，应**中止并报错**，而不是放行——否则任何一处业务查询忘带 ctx 都会变成权限漏洞。
2. **字段屏蔽要兜底**：`db.Omit()` 只在 struct 模型查询下生效，遇到 `Select("*")` 或原生 SQL 会失效。敏感字段建议在响应序列化前再过滤一次，不单靠 Omit。
3. **domain 必须显式注册**：`resolveDomain` 只查 `domain.go` 里显式登记的 表名→domain 映射，不自动发现；join / 子查询 / alias 等场景取不到表名时按"无规则"处理并记录告警。
4. **缓存必须有失效机制**：`engine` 缓存规则结果，管理端增删改 `sys_rule` / `sys_rule_assignment` 时主动清缓存（或用版本号），否则规则改了不生效。

| `/api/role/list` | GET | 角色列表（查 sys_role） |

| `/api/role/delete` | POST | 删除角色（删 sys_role 一行 + 清除 Casbin 该 role_code 的全部 p/g，默认批量） |

| `/api/role/user/list` | GET | 角色的用户列表（g 映射反查） |
| `/api/role/user/save` | POST | 保存角色的用户（写入 Casbin g 策略） |

## 十二、边界说明

- 数据权限默认只影响 LIST/分页查询，不影响 CREATE/UPDATE/DELETE 的权限控制
- Casbin policy 只存 `allow` 规则
- 数据域 domain 需要在代码中显式注册，不自动发现
- 公共接口（无需登录即可访问，如 `/api/captcha`、`/livez`、`/readyz`）通过中间件路由分组（免 JWT 组）或白名单处理，不走 Casbin 校验
- `sys_menus.is_public` 仅控制"菜单是否无需权限可见"（已登录用户侧的菜单渲染），与"API 是否免登录"是两个概念，不要混用
- Casbin sub 约定：用户侧用 `user_id`（数字字符串），角色侧用 `role_code`（字符串）；角色元信息存 `sys_role`，用户↔角色、角色↔权限关系都走 Casbin（g / p），不建 sys_admin_role
