# 权限管理开发方案

更新时间：2026-05-09

---

## 一、目标

基于 Casbin 构建权限系统，覆盖三个层次：

```
Layer 1: API 权限     — 能不能访问这个接口？
Layer 2: 行级数据权限  — 能看见哪些数据行？（时间范围、部门范围等）
Layer 3: 字段级权限   — 能看见哪些数据列？（敏感字段屏蔽）
```

默认走角色（RBAC），支持用户级例外（允许额外权限 / 禁止特定权限）。

---

## Casbin 管什么，新增表管什么

Casbin 只做一件事：**判断"谁(sub)能不能对什么资源(obj)做什么操作(act)"**，返回 true/false。

以下场景 Casbin **不做**，需要新增表和逻辑：

| 场景 | Casbin 为什么做不了 | 谁来处理 |
|------|--------------------|---------|
| URL path → perm_code 翻译 | Casbin 不认识 HTTP URL，它只认策略里写的 obj | Casbin 直接存 URL（v1），code 放 v3 做关联，不需要额外映射表 |
| 张三在角色基础上多一个权限 | Casbin 策略是"角色→权限"，没有"角色+例外"概念 | `sys_user_permission_exception` 表 |
| 订单列表只显示最近 30 天 | Casbin 不参与 SQL 构建，不做行级过滤 | `sys_data_rule` 表 + GORM 插件 |
| 金额字段对某些人不可见 | Casbin 不知道"字段"这个概念 | `sys_field_rule` 表 + 响应后处理 |
| 某个接口所有用户都能访问 | 理论上可以 Casbin 放行，但更轻的方式是标记 | `sys_menus.is_public = 1` |

## 二、整体架构

```
请求 → JWT 认证 → 获取 user_id + role_list
    │
    ├── Layer 1: API 权限（Casbin）
    │   请求 URL + Method
    │   → Casbin enforce(user.roles, URL_PATH, METHOD)
    │   → 查 sys_user_permission_exception（用户例外覆盖）
    │   → 通过 / 拒绝
    │
    └── Layer 2: 数据权限（GORM Plugin + JSqlParser）
         业务 GORM 查询
         → GORM BeforeQuery 拦截器
         → 检测数据域 domain
         → 加载 sys_data_rule JSON 规则
         → JSqlParser 翻译为 SQL WHERE 条件
         → 注入原始 SQL
         → 执行过滤后的 SQL
```

## 三、数据流

### 3.1 API 权限校验流

```
┌─────────────┐
│  HTTP 请求   │  GET /api/admin/list
└──────┬──────┘
       │
       ▼
┌─────────────────────┐
│  JWT 鉴权中间件       │  提取 user_id, role_ids
└──────┬──────────────┘
       │
       ▼
┌──────────────────────────────┐
│  Casbin 角色权限校验          │
│  enforce(role_ids, '/api/admin/list', 'GET')  │
│  │                             │
│  ├─ 通过 ──→ 继续             │
│  └─ 拒绝 ──→ 查用户例外        │
└──────┬───────────────────────┘
       │
       ▼
┌──────────────────────────────────────┐
│  查 sys_user_permission_exception    │
│  WHERE admin_id = ? AND perm_code = ?│
│  （从 Casbin 策略中查出该 URL 对应     │
│   的 code，再用 code 查例外表）        │
│  │                                   │
│  ├─ effect=allow ──→ 放行（覆盖角色拒绝）│
│  ├─ effect=deny  ──→ 拒绝（覆盖角色允许）│
│  └─ 无记录 ──→ 以角色校验结果为准       │
└──────┬──────────────────────────────┘
       │
       ▼
    通过 / 拒绝
```

## 四、数据库表结构

### 4.1 已有表（保留不动）

| 表名 | 用途 | 说明 |
|------|------|------|
| `sys_admin` | 管理员用户 | 已有 |
| `sys_role` | 角色定义 | 已有，只关联权限 |
| `sys_admin_role` | 用户↔角色关联 | g.admin_id, role_id |
| `sys_menus` | 菜单树 | 保留 permission_code 关联权限 |
| `sys_menu_permission` | 菜单↔权限关联 | 已有 |
| `sys_casbin_rule` | Casbin 策略 | v0=角色, v1=URL, v2=方法, v3=code，自动管理 |

### 4.2 新增表

新增三张表，分工如下：

```
sys_user_permission_exception  →  用户级 API 权限例外（Casbin 管不到的单用户覆盖）
sys_data_rule                  →  行级数据过滤规则（Casbin 管不到的行级 SQL 注入）
sys_field_rule                 →  字段级可见性规则（Casbin 管不到的列级屏蔽）
```

#### sys_user_permission_exception — 用户级权限例外

**解决的问题**：用户 A 属于"编辑"角色，编辑角色没有"报表查看"权限，但用户 A 需要额外有这个权限。
或者反过来：用户 B 属于"管理员"角色，管理员可以删除数据，但用户 B 不应该能删除。

Casbin 做不到"角色 + 例外"，因为 Casbin 的 RBAC 模型里，权限只挂在角色上。

```sql
CREATE TABLE `sys_user_permission_exception` (
  `id` bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `admin_id` bigint unsigned NOT NULL COMMENT '用户ID',
  `perm_code` varchar(100) NOT NULL COMMENT '权限编码',
  `effect` varchar(10) NOT NULL DEFAULT 'allow' COMMENT 'allow 放行 / deny 禁止',
  `create_by` bigint unsigned DEFAULT NULL COMMENT '创建人ID',
  `create_time` datetime(3) DEFAULT NULL COMMENT '创建时间',
  `update_by` bigint unsigned DEFAULT NULL COMMENT '更新人ID',
  `update_time` datetime(3) DEFAULT NULL COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_admin_perm` (`admin_id`, `perm_code`),
  KEY `idx_admin_id` (`admin_id`)
) ENGINE=InnoDB COMMENT='用户权限例外表';
```

使用示例：

```
# 用户 3 属于 editor 角色，editor 不能删除，但单独给 3 开删除权限
INSERT INTO sys_user_permission_exception VALUES (admin_id=3, perm_code='admin:delete', effect='allow');

# 用户 1 属于 admin 角色，admin 能删除，但不允许 1 删除
INSERT INTO sys_user_permission_exception VALUES (admin_id=1, perm_code='admin:delete', effect='deny');
```

#### sys_rule — 数据权限规则模板

**解决的问题**：统一控制数据访问权限，包含两种能力：
- **字段屏蔽**：用户不能查看某些字段（如金额、利润率）
- **内容过滤**：用户不能查看包含特定内容的行（如标题含"折扣"）

规则模板存一次，通过分配表绑定到多个角色或用户。

```sql
CREATE TABLE `sys_rule` (
  `id` bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `rule_name` varchar(100) NOT NULL COMMENT '规则名称',
  `domain` varchar(50) NOT NULL COMMENT '数据域标识（ORDERS / NOTICE / ADMIN 等）',
  `config` json NOT NULL COMMENT '规则配置JSON（含 omit_fields + conditions）',
  `status` tinyint NOT NULL DEFAULT '1' COMMENT '状态: 0=禁用 1=启用',
  `remark` varchar(200) DEFAULT NULL COMMENT '备注',
  `create_by` bigint unsigned DEFAULT NULL COMMENT '创建人ID',
  `create_time` datetime(3) DEFAULT NULL COMMENT '创建时间',
  `update_by` bigint unsigned DEFAULT NULL COMMENT '更新人ID',
  `update_time` datetime(3) DEFAULT NULL COMMENT '更新时间',
  PRIMARY KEY (`id`),
  KEY `idx_domain` (`domain`),
  KEY `idx_status` (`status`)
) ENGINE=InnoDB COMMENT='数据权限规则表';
```

**sys_rule_assignment** — 规则分配

```sql
CREATE TABLE `sys_rule_assignment` (
  `id` bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `rule_id` bigint unsigned NOT NULL COMMENT '规则ID',
  `target_type` tinyint NOT NULL COMMENT '目标类型: 1=角色 2=用户',
  `target_id` bigint unsigned NOT NULL COMMENT '目标ID（role_id 或 admin_id）',
  `create_by` bigint unsigned DEFAULT NULL,
  `create_time` datetime(3) DEFAULT NULL,
  PRIMARY KEY (`id`),
  KEY `idx_rule_id` (`rule_id`),
  KEY `idx_target` (`target_type`, `target_id`),
  CONSTRAINT `fk_rule_assign` FOREIGN KEY (`rule_id`) REFERENCES `sys_rule` (`id`) ON DELETE CASCADE
) ENGINE=InnoDB COMMENT='规则分配表';
```

##### config JSON 格式

config 字段统一包含 `omit_fields`（字段屏蔽）和 `condition_groups`（内容过滤条件组），两者可同时存在，也可只用一个。

每组有自己的 `logic`（AND/OR），组之间默认 AND 连接。通过多组组合，可以实现复杂的过滤逻辑，避免全 AND 把所有数据都筛掉。

```json
{
  "omit_fields": ["amount", "profit_rate"],
  "condition_groups": [
    {
      "logic": "AND",
      "conditions": [
        { "field": "dept_id", "op": "EQ", "value": "user.dept_id", "desc": "只能看本部门" },
        { "field": "status", "op": "NOT_IN", "value": ["deleted"], "desc": "不能看已删除" }
      ]
    },
    {
      "logic": "OR",
      "conditions": [
        { "field": "title", "op": "NOT_LIKE", "value": "%折扣%", "desc": "标题不能含折扣" },
        { "field": "title", "op": "NOT_LIKE", "value": "%特价%", "desc": "标题不能含特价" }
      ]
    }
  ]
}
```

生成的 SQL：

```sql
SELECT id, title, status, dept_id   -- amount 被 Omit
FROM orders
WHERE (dept_id = 用户部门 AND status != 'deleted')
  AND (title NOT LIKE '%折扣%' OR title NOT LIKE '%特价%')
```

支持的 op 操作符：`EQ`, `NEQ`, `GT`, `GTE`, `LT`, `LTE`, `IN`, `NOT_IN`, `LIKE`, `NOT_LIKE`, `BETWEEN`

值引用：`user.*` 前缀表示从用户上下文取动态值（如 `user.dept_id`）；直接写值表示固定值。

## 五、前端权限展示设计

### 5.1 权限在管理界面中的展示形式

给角色分配权限时，前端展示两段内容：

#### 5.1.1 菜单+按钮树（核心展示区）

只显示 `sys_menu_permission` 中有关联记录的权限，在 `sys_menus` 中按 type 分层展示：

```
type 1 = 目录（分组，无页面，不可勾选，只做容器）
type 2 = 菜单（有页面，可导航，可勾选）
type 3 = 按钮（页面内的操作按钮，处于对应菜单下，可勾选）

示例：
系统管理 (type=1, 目录)
├── ☐ 用户管理 (type=2, 菜单 → perm_code: admin:list)
│   ├── ☐ 新增 (type=3, 按钮 → perm_code: admin:create)
│   ├── ☐ 编辑 (type=3, 按钮 → perm_code: admin:update)
│   └── ☐ 删除 (type=3, 按钮 → perm_code: admin:delete)
├── ☐ 角色管理 (type=2, 菜单 → perm_code: role:list)
│   └── ☐ 分配权限 (type=3, 按钮 → perm_code: role:assign)
└── ☐ 操作日志 (type=2, 菜单 → perm_code: log:list)
```

判断依据：`sys_menus.id IN (SELECT menu_id FROM sys_menu_permission)`

- 有关联的菜单/按钮 → 展示在角色分配界面上
- 没有关联的菜单/按钮 → 不在权限分配界面出现

#### 5.1.2 独立权限列表（无对应菜单的后端权限）

后端 API 权限（定时任务、数据同步、内部回调等）没有对应菜单，但通过 Casbin 策略注册了 code，作为独立列表展示在角色分配界面底部：

```
独立权限（无对应菜单）：
☐ 数据同步 (perm_code: sync:data)
☐ 定时任务触发 (perm_code: cron:trigger)
☐ 批量导入 (perm_code: import:batch)
☐ 回调通知 (perm_code: callback:receive)
```

**筛选规则**：`Casbin 策略中的 code NOT IN (SELECT permission_code FROM sys_menu_permission)`

### 5.2 角色绑菜单→权限落地方案

```
前端操作：角色 ←→ 勾选菜单/按钮树
              │
              ▼
后端翻译：被勾选的菜单/按钮
  → 查 sys_menu_permission 拿到所有关联的 perm_code
  → 查每个 perm_code 对应的 URL 和 Method（从菜单绑定的路由信息获取）
  → 去重合并
  → 写入 Casbin policy: p, role_code, URL, METHOD, code
              │
              ▼
Casbin 校验：请求 URL + Method
  → enforce(role, URL, method)
  → 不经过任何映射，直接命中策略
```

**存储示例（sys_casbin_rule）：**

```
ptype  v0         v1                   v2    v3
p      admin      /api/admin/list      GET   admin:list
p      editor     /api/admin/list      GET   admin:list
p      admin      /api/admin/create    POST  admin:create
```

**核心原则**：
- **URL（v1）+ 方法（v2）**：参与 enforce，用于请求时鉴权
- **code（v3）**：不参与 enforce，只做关联 key，用于菜单渲染、角色绑定、用户例外

### 5.3 用户渲染菜单树

```
用户登录
  → 查 Casbin 策略（通过角色查找所有 p 规则）
  → 拿到 p 规则中 v3（code）列表
  → 合并 sys_user_permission_exception（effect=allow）的 code
  → 查 sys_menu_permission 拿 menu_id 列表
  → 查 sys_menus 组装树（过滤 type=2,3）
  → 返回前端渲染
```

用户独有的 `sys_user_permission_exception`（effect=allow）追加到 code 列表中一同处理。

### 5.4 用户额外权限界面

用户管理 → 选择用户 → "额外权限" tab：
- 展示同角色分配界面的菜单+按钮树
- 只显示该用户当前角色**没有**的权限（管理员只需关心额外开的）
- 勾选 → 写入 `sys_user_permission_exception`

## 六、Casbin 策略模型

### 6.1 模型定义

Casbin 模型改为 4 字段 `p = sub, obj, act, code`，code 不参与 matcher：

```
[request_definition]
r = sub, obj, act

[policy_definition]
p = sub, obj, act, code

[role_definition]
g = _, _

[policy_effect]
e = some(where (p.eft == allow))

[matchers]
m = g(r.sub, p.sub) && r.obj == p.obj && r.act == p.act
```

matcher 不变，code 不参与 enforce。

### 6.2 Policy 策略

```
# 角色权限：p, role_code, URL, METHOD, code
p, admin, /api/admin/list,   GET,  admin:list
p, admin, /api/admin/create, POST, admin:create
p, editor, /api/admin/list,  GET,  admin:list

# 用户↔角色关联（与 sys_admin_role 同步写入）：
g, 1, admin
g, 2, editor
```

### 6.3 中间件实现

```
obj := c.Request.URL.Path           // "/api/admin/list"
act := c.Request.Method              // "GET"
sub := strconv.FormatInt(userID, 10)
ok, err := enforcer.Enforce(sub, obj, act)
```

## 七、pkg 层新增组件

### pkg/datarule — 数据权限引擎

```
pkg/datarule/
├── datarule.go           # 组件入口 Init / Close / Ready
├── types.go              # 规则相关类型定义
├── engine.go             # 加载规则、匹配用户、缓存
├── injector.go           # 条件注入：conditions → GORM WHERE
├── omit.go               # 字段排除：omit_fields → GORM Omit
├── domain.go             # 数据域注册：表名 ↔ domain 映射
└── provider/
    └── gorm.go           # GORM 插件（BeforeQuery 拦截器）
```

#### 核心实现

**数据加载流程（engine.go）：**

```
用户查询某张表时：
  1. 确定 domain（通过表名）
  2. 查 sys_rule_assignment，找到该用户/角色被分配了哪些规则：
     SELECT rule_id FROM sys_rule_assignment
     WHERE (target_type=1 AND target_id IN role_ids)
        OR (target_type=2 AND target_id = admin_id)
  3. 根据 rule_id 加载规则模板：
     SELECT config FROM sys_rule WHERE id IN (rule_ids)
  4. 缓存结果
```

**GORM 插件（provider/gorm.go）：**

```go
type DataRulePlugin struct{}

func (p *DataRulePlugin) Name() string {
    return "DataRulePlugin"
}

func (p *DataRulePlugin) Initialize(db *gorm.DB) error {
    return db.Callback().Query().Before("gorm:query").
        Register("data_rule:inject", p.inject)
}

func (p *DataRulePlugin) inject(db *gorm.DB) {
    ctx := db.Statement.Context
    if ctx == nil {
        return
    }
    adminID, _ := ctx.Value("admin_id").(int64)
    roleIDs, _ := ctx.Value("role_ids").([]int64)
    if adminID == 0 {
        return
    }

    domain := resolveDomain(db.Statement.Table)

    // 查分配表 → 取 rule_id 列表 → 加载统一规则 config
    rules := engine.LoadRules(domain, adminID, roleIDs)
    for _, rule := range rules {
        // 字段屏蔽
        if len(rule.Config.OmitFields) > 0 {
            db.Omit(rule.Config.OmitFields...)
        }
        // 内容过滤：按条件组处理，每组有自己的 logic
        for _, group := range rule.Config.ConditionGroups {
            if len(group.Conditions) > 0 {
                injector.ApplyGroup(db, group)
            }
        }
    }
}
```

一条规则同时处理字段屏蔽和内容过滤，不需要分两张表查两次。

```sql
-- 第 1 步：建规则模板

  'NOTICE',
  '{"omit_fields":["is_popup"]}',
  1
);

-- 第 2 步：分配给用户 3

-- 第 3 步：也分配给角色 2
```

## 十二、边界说明

- 数据权限默认只影响 LIST/分页查询，不影响 CREATE/UPDATE/DELETE 的权限控制
- 用户例外优先级：`sys_user_permission_exception.effect=deny` > `Casbin policy allow`
- Casbin policy 只存 `allow` 规则，不存 `deny` 规则（`deny` 通过例外表实现）
- 数据域 domain 需要在代码中显式注册，不自动发现
- 公共接口（无需登录即可访问）通过 `sys_menus.is_public = 1` 标记，不走 Casbin 校验
